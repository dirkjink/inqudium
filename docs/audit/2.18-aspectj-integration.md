# 2.18 AspectJ Integration Audit

**Scope:** Phase-2 sub-step 2.18 of `REFACTORING.md`. The first of the three integration audits
that follow ADR-033 closure. Verifies that the AspectJ integration in `inqudium-aspect` and
`inqudium-aspect-integration-tests` is consistent with the new bulkhead architecture
(`InqBulkhead<A, R>` as `BulkheadHandle<ImperativeTag>`, `InqElement`, `InqDecorator<A, R>`).

**Method:** Inventory, then assessment against the four audit areas (2.18.2 — `InqBulkhead`
acceptance, 2.18.3 — lifecycle-phase compatibility, 2.18.4 — test coverage). No code changes.

**Build status:** `mvn verify` across the full reactor — green.
Aspect modules: `inqudium-aspect` + `inqudium-aspect-integration-tests`, 134 tests, 0 failures,
0 errors, 0 skipped (sum of nested-class surefire reports — outer classes report 0 because
`@Nested` is in use).

## 2.18.2 — `InqBulkhead` acceptance

For each component that has a type bound or eager validation involving `InqDecorator` or `InqAsyncDecorator`, can it accept a `Inqudium.configure()`-built `InqBulkhead`?

| Component | Sync acceptance (`InqDecorator<A, R>`) | Async acceptance (`InqAsyncDecorator<A, R>`) | Verdict |
|-----------|-----------------------------------------|------------------------------------------------|---------|
| `ElementLayerProvider` | Accepts `InqBulkhead<Void, Object>` (intersection `<E extends InqElement & InqDecorator<Void, Object>>` is satisfied by ADR-033 Stage 3). Pinned by `Adr033Stage3ClosureTest`. | n/a (sync only) | **kompatibel** |
| `AsyncElementLayerProvider` | n/a (async only) | Bound `<E extends InqElement & InqAsyncDecorator<Void, Object>>` is **not** satisfied by `InqBulkhead` — the class intentionally does not implement `InqAsyncDecorator` (an async `InqBulkhead` variant is deferred to its own ADR per `InqBulkhead` Javadoc and `TODO` line 195). Today only the deprecated `Bulkhead`/`ImperativeBulkhead` interfaces satisfy this bound. | **inkompatibel** (async path) |
| `AspectPipelineTerminal` (delegates to `SyncPipelineTerminal`) | Pipeline elements must implement `InqDecorator`. `InqBulkhead` does. | n/a | **kompatibel** |
| `HybridAspectPipelineTerminal` | Pipeline elements must implement **both** `InqDecorator` **and** `InqAsyncDecorator`, validated eagerly in the constructor. `InqBulkhead` does not implement `InqAsyncDecorator`, so a pipeline that includes an `InqBulkhead` throws `ClassCastException` at `of(...)` — even when only sync methods will ever be intercepted. | **inkompatibel** (eager-validation block) |
| `AspectPipelineBuilder` / `ResolvedPipeline` / `AbstractPipelineAspect` | These are `LayerAction`-based, not `InqElement`-based. They accept any provider, which the user wires via `ElementLayerProvider`. Routing is identical for `InqBulkhead` and any other element. | — (sync side) | **kompatibel** |
| `AsyncAspectPipelineBuilder` / `AsyncResolvedPipeline` / `AbstractAsyncPipelineAspect` | — | Same. The bottleneck is upstream in `AsyncElementLayerProvider`. | **kompatibel** (downstream) |

**Entry-point usage of paradigm-specific APIs:** none. Every component contracts against the
pipeline `LayerAction` / `AsyncLayerAction` / `InqDecorator` / `InqAsyncDecorator` types. There is
no call to `bulkhead.execute(...)` or `bulkhead.tryAcquire(...)` anywhere in the aspect modules.
This is the right shape for paradigm-agnostic integration.

**Inheritance of `InqElement` accessors:** `ElementLayerProvider` derives `layerName` from
`element.elementType().name()` + `element.name()`. `HybridAspectPipelineTerminal` does the same in
its diagnostic state. `eventPublisher()` is **not** consulted by any aspect-side code — the layer
emits no events of its own; observability is the bulkhead's responsibility (per ADR-030 and the
component-publisher factory wiring inside `InqBulkhead`).

## 2.18.3 — Lifecycle-phase compatibility

The relevant aspect path on every invocation is:

```
@Around → executeAround(pjp)
       → ResolvedPipeline.execute (or HybridAspectPipelineTerminal.execute*Chain)
            → composed inline chain
                 → LayerAction.execute(c, ca, a, next)        ← layer N
                      → ...
                      → LayerAction.execute(c, ca, a, next)   ← bulkhead layer
                            → InqBulkhead.execute(c, ca, a, next)         (inherited final)
                                 → ImperativeLifecyclePhasedComponent.execute
                                      → phase.get()    (AtomicReference, volatile read)
                                      → phase.execute(c, ca, a, next)
                                           → BulkheadHotPhase.execute → strategy (volatile)
```

The `LayerAction` reference cached by `ElementLayerProvider` (`element::execute`) and by
`HybridAspectPipelineTerminal` (the `InqDecorator` is the `LayerAction`, since `InqDecorator extends
LayerAction`) is the `InqBulkhead` instance itself, not a snapshot of phase or strategy. Each
aspect invocation therefore re-reads `phase` (`AtomicReference#get()`) and, inside the hot phase,
the volatile `strategy` field.

| Lifecycle event | Aspect-path behaviour |
|-----------------|------------------------|
| Cold-to-hot transition | First aspect call hits `InqBulkhead.execute` → `phase.get()` returns `ColdPhase` → the cold phase runs the CAS, fires `ComponentBecameHotEvent`, delegates to the just-installed hot phase. Subsequent aspect calls observe `HotPhase` directly. **Compatible.** No cached pre-hot reference exists in any aspect component. |
| Strategy hot-swap (ADR-032) | Snapshot listener installed by `BulkheadHotPhase.afterCommit` writes the new `BulkheadStrategy` to the volatile `strategy` field. The next aspect call enters `BulkheadHotPhase.execute`, which re-reads `strategy`. **Compatible.** Aspect provider does not cache a `BulkheadStrategy` or a `BulkheadHotPhase` reference. |
| Removal (`markRemoved()`) | `phase` becomes `RemovedPhase` sentinel. Next aspect call: `ImperativeLifecyclePhasedComponent.execute` checks `if (current instanceof RemovedPhase) throw new ComponentRemovedException(...)`. The exception propagates back through the layer chain to `executeAround` and out through the `@Around` advice. **Compatible.** AspectJ surfaces the original throwable type unchanged. |

**Caching surfaces inspected, no cached lifecycle state found:**

- `ElementLayerProvider`: caches `element`, `layerAction = element::execute`, `order`, `layerName`. No phase or strategy reference.
- `AsyncElementLayerProvider`: caches `element`, `asyncLayerAction = element::executeAsync`, `order`, `layerName`. (Not currently reachable for `InqBulkhead`; see 2.18.2.)
- `HybridAspectPipelineTerminal`: caches `syncActions[]`, `asyncActions[]` (the decorators themselves), `pipelineState`, per-`Method` async-flag map. The decorators are stable references; per-call lifecycle indirection happens inside the decorator.
- `ResolvedPipeline` / `AsyncResolvedPipeline`: cache `LayerAction[]` / `AsyncLayerAction[]` extracted from providers. Same — the action *is* a method reference into the live element.
- `AbstractPipelineAspect` / `AbstractAsyncPipelineAspect`: cache the `ResolvedPipeline` per `Method`. The pipeline's actions point at the providers' actions which point at the elements.

**Conclusion:** the aspect pipeline routes every invocation through a method reference into
`InqBulkhead`, which then reads volatile state per call. Nothing is cached at a level below the
element's own indirection. The 2.18.3 audit area is structurally clean.

## 2.18.4 — Test coverage

Per the inventory in 2.18.1, exactly one aspect test exercises a real `InqBulkhead` —
`Adr033Stage3ClosureTest` — and it pins only the structural fact that `ElementLayerProvider`'s
intersection bound accepts the type. There is no aspect test that:

- Drives a real bulkhead through a cold-to-hot transition via an aspect-mediated invocation.
- Triggers a strategy hot-swap on a real bulkhead and observes that the next aspect-mediated call
  uses the new strategy.
- Calls a real bulkhead through an aspect after `markRemoved()` and asserts
  `ComponentRemovedException`.
- Wraps a real bulkhead in `AspectPipelineTerminal` (composition-style) and exercises the chain.
- Covers the `HybridAspectPipelineTerminal` sync-only path with a real bulkhead — currently
  blocked by the eager-validation finding.

All other aspect tests (`AspectPipelineTerminalTest`, `HybridAspectPipelineTerminalTest`,
`AspectPipelineBuilderTest`, `AbstractPipelineAspectTest`, `AbstractAsyncPipelineAspectTest`) use
synthetic `LayerAction` lambdas or dedicated stub decorators (`TracingDecorator`, dual-stub
decorators). The integration tests (`AspectJCtwIntegrationTest`, `PerLayerAnnotationTest`) exercise
CTW weaving end-to-end but use cross-cutting layers (Logging/Authorization/Timing), not resilience
elements. No integration test wires up a `Inqudium.configure()`-built bulkhead.

This is the same shape as the 2.17.4 finding for the wrapper layer: pre-ADR-033 the design forced
synthetic stubs because real bulkheads did not satisfy the `InqDecorator` bound. Stage 3 closed the
structural gap. 2.18.4 is the corresponding test gap on the aspect side.

The structural compatibility shown by 2.18.2 and 2.18.3 makes these tests writeable today — the
absence is a routing question, not a blocker.

## Findings

### F-2.18-1 — `HybridAspectPipelineTerminal` rejects pipelines containing `InqBulkhead`

- **Bereich:** 2.18.2 (acceptance).
- **Priorität:** kritisch (für die hybride Variante; nicht-blockierend für aktuelle Nutzer).
- **Beobachtung:** `HybridAspectPipelineTerminal.of(InqPipeline)` validiert eager im Konstruktor,
  dass jedes Pipeline-Element **sowohl** `InqDecorator` **als auch** `InqAsyncDecorator`
  implementiert (Zeilen 152–157 + Helpers `asDecorator`/`asAsyncDecorator`). `InqBulkhead`
  implementiert nur `InqDecorator` — der asynchrone Vertrag ist absichtlich nicht implementiert
  (Javadoc Zeile 46–47, `TODO` Zeile 195: "an asynchronous variant of InqBulkhead — analogous to
  the old `InqAsyncDecorator`"). Folge: jedes `InqPipeline` mit einem `InqBulkhead` wirft
  `ClassCastException` beim `of(...)`-Aufruf, auch dann, wenn die intercepted Methode synchron ist
  und der hybride Async-Pfad zur Laufzeit nie betreten würde.
- **Begründung der Priorität:** kritisch *für die Hybrid-Variante*, weil sie die einzige
  Komposition-basierte Variante ist, die sync und async im selben Aspekt vereinen kann. Aber:
  `AspectPipelineTerminal` (sync-only) und `AbstractPipelineAspect` mit `ElementLayerProvider`
  bieten beide funktionierende Pfade für rein synchrone Aspekte mit Bulkhead. Ein sofort
  blockendes Problem ist es nur dann, wenn ein Nutzer zwingend Hybrid-Dispatch wollte.
- **Routing-Vorschlag:** Zwei Optionen, beide in einem späteren Sub-Step adressierbar.
  1. *Lockern der Eager-Validation:* `HybridAspectPipelineTerminal` sollte den `InqAsyncDecorator`-
     Cast pro Element nur dann erzwingen, wenn dieses Element tatsächlich für eine
     `CompletionStage`-Methode angefordert wird. Konkret: `asyncActions` lazy aufbauen oder
     Elemente, die nur `InqDecorator` sind, dürfen sync-only beitragen, und der Async-Pfad meldet
     für solche Elemente einen klaren Fehler erst dann, wenn er ausgeführt wird. Diese Lockerung
     hat Designkonsequenzen — sollte als Sub-Step (z. B. "2.18.5 — Hybrid-Terminal-Lockerung der
     Eager-Validation") oder via dedizierter ADR diskutiert werden.
  2. *Async-`InqBulkhead`-Variante einführen* (das ohnehin angekündigte
     `TODO` aus `InqBulkhead.java:195`). Dann erfüllt `InqBulkhead` beide Verträge und
     `HybridAspectPipelineTerminal` akzeptiert ihn ohne Sonderbehandlung. Das ist die größere
     Arbeit und gehört unter ein eigenes ADR.
  Empfehlung: Finding nach ADR-Audit-Backlog oder `TODO.md` aufnehmen, kein eigener Sub-Step
  innerhalb dieses Bulkhead-Refactors. Der aktuelle Refactor-Scope ist ohnehin auf den
  imperativen, synchronen Bulkhead beschränkt.

### F-2.18-2 — `AsyncElementLayerProvider` ist mit `InqBulkhead` nicht konstruierbar

- **Bereich:** 2.18.2 (acceptance, async).
- **Priorität:** wichtig (für den async-Pfad; nicht-blockierend für aktuelle Nutzer).
- **Beobachtung:** Konstruktor-Bound `<E extends InqElement & InqAsyncDecorator<Void, Object>>`
  (Zeilen 80, 102, 123) lehnt `InqBulkhead` zur Compile-Zeit ab. Heute erfüllt nur das
  deprecated `Bulkhead`/`ImperativeBulkhead`-Paar diesen Bound. Folge: Bulkhead-Aspect-Integration
  über den asynchronen Pfad ist an deprecated Typen gebunden, bis eine async-`InqBulkhead`-Variante
  existiert.
- **Begründung der Priorität:** wichtig, nicht kritisch — die synchrone Bulkhead-Aspect-Integration
  funktioniert (`ElementLayerProvider`), und der Refactor-Scope deckt async ohnehin erst über
  einen separaten ADR ab. Aber: jeder Async-Aspekt-Setup, der heute mit dem deprecated `Bulkhead`
  arbeitet, kann nicht migriert werden, ohne den Async-Pfad aus dem Aspekt zu entfernen. Das
  bremst die Ablösung der deprecated Typen.
- **Routing-Vorschlag:** identisch zu F-2.18-1: an den ADR-Audit/`TODO.md` weitergeben, eigentliche
  Auflösung erfolgt mit dem async-`InqBulkhead`-ADR. Kein eigener Sub-Step im aktuellen Refactor.
  Wenn ein Sub-Step gewünscht ist, dann gemeinsam mit F-2.18-1 ("Async-Bulkhead und
  Aspect-Integration") als ein Block.

### F-2.18-3 — Aspect-Tests gegen echte `InqBulkhead` fehlen

- **Bereich:** 2.18.4 (test coverage).
- **Priorität:** wichtig (analog zu Audit-Finding 2.17.4).
- **Beobachtung:** Außer `Adr033Stage3ClosureTest` (struktureller Compile-Beweis) exerziert kein
  Aspect-Test einen `Inqudium.configure()`-gebauten Bulkhead. Synthetische `LayerAction`-Lambdas
  und Stub-Decorator (`TracingDecorator`, Dual-Stub) sind die Norm. Die in 2.18.3 strukturell
  validierten Lifecycle-Eigenschaften (cold-to-hot, hot-swap, removal) sind end-to-end durch den
  Aspekt-Pfad nicht gepinnt.
- **Begründung der Priorität:** wichtig, weil:
  1. Der strukturelle Beweis in 2.18.3 ist ein Code-Review-Argument, kein Regressionsschutz. Eine
     Refaktorisierung an `BulkheadHotPhase` oder `ImperativeLifecyclePhasedComponent` könnte
     unbemerkt das Lifecycle-Verhalten durch den Aspekt-Pfad brechen.
  2. ADR-033 hat die strukturelle Voraussetzung (`InqBulkhead` ⇒ `InqDecorator`) gerade erst
     geschaffen — die Mock-Tendenz aus der Pre-ADR-033-Ära ist Pragmatismus, nicht Designprinzip.
  3. Die Test-Form ist analog zur 2.17.4-Beobachtung; wenn 2.20 ohnehin Wrapper-und-Proxy-Tests
     gegen echte Bulkheads aufnimmt, ist die Inklusion der Aspect-Pfade kostengünstig.
- **Routing-Vorschlag:** in 2.20 (`inqudium-bulkhead-integration-tests`) absorbieren, parallel zum
  bereits dort gelisteten "Wrapper compatibility across cold-to-hot transition, strategy
  hot-swap, and structural removal" und den "Wrapper and proxy tests run against real bulkheads"
  Findings. Konkrete neue Test-Klassen in dem Modul:
  - `BulkheadAspectLifecycleTest` — drei Szenarien analog zur Wrapper-Compat-Liste:
    A) cold-to-hot durch `@Around`-Aufruf,
    B) strategy hot-swap reflected im nächsten Aspekt-Aufruf,
    C) `markRemoved()` löst `ComponentRemovedException` durch den Aspekt-Pfad aus.
  - `BulkheadAspectTerminalTest` — gleicher Szenarien-Satz für `AspectPipelineTerminal` (sync-only,
    da `HybridAspectPipelineTerminal` durch F-2.18-1 blockiert ist).
  Voraussetzung: das 2.20-Modul muss Maven-Abhängigkeiten zu `inqudium-aspect`,
  `inqudium-imperative`, `inqudium-config` aufnehmen. Falls 2.20 reine
  Integration-Test-Variant-by-variant geplant ist (Wrapper-only, Spring-only, Aspect-only), gehört
  ein `aspect/`-Unterpaket in das Modul.

### F-2.18-4 — Veraltete Konstruktor-Beispiele in der Javadoc von `ElementLayerProvider` und `AsyncElementLayerProvider`

- **Bereich:** Drift / Doku-Konsistenz.
- **Priorität:** nachrangig.
- **Beobachtung:** Beide Klassen-Javadocs zeigen Beispiele wie
  `ImperativeBulkhead<Void, Object> bulkhead = new ImperativeBulkhead<>(cfg, strategy);` und
  `ImperativeRetry<Void, Object> retry = new ImperativeRetry<>(retryCfg);`
  (`ElementLayerProvider.java:37-39`, `AsyncElementLayerProvider.java:34-35`). Die `cfg`/`strategy`-
  Konstruktoren existieren in der neuen Architektur nicht; `Inqudium.configure()...build()` ist
  der einzige Pfad, einen `InqBulkhead` zu materialisieren. `ImperativeRetry` existiert ohnehin
  noch nicht (Retry steht außerhalb des aktuellen Bulkhead-Refactors).
- **Begründung der Priorität:** nachrangig — die Javadoc-Beispiele dokumentieren ein API, das
  für `InqBulkhead` so nie existiert hat (Stage 3 ist der erste Punkt, an dem `InqBulkhead` als
  Provider-Argument funktioniert). Sie verwirren neue Leser, blockieren aber niemanden. Die
  konstruktive Gestaltung der Beispiele kann erst sinnvoll erfolgen, sobald eine Codestelle
  existiert, die sie spiegeln (etwa `BulkheadAspectLifecycleTest` aus F-2.18-3).
- **Routing-Vorschlag:** Doku-Korrektur als trivialer Edit zulässig (Javadoc-Beispiele auf
  `Inqudium.configure().imperative(im -> im.bulkhead("name", b -> b.balanced())).build()`-Idiom
  umschreiben). Nicht jetzt vornehmen, sondern *gemeinsam* mit F-2.18-3 in 2.20 — dann hat man eine
  echte Test-Klasse zum Verlinken und braucht nicht zweimal über die richtige Code-Form
  nachzudenken. Alternativ als `TODO.md`-Eintrag `TODO(2.20): Javadoc-Beispiele in
  ElementLayerProvider/AsyncElementLayerProvider auf Inqudium.configure() umstellen, sobald die
  Aspect-Integration-Tests Referenzcode liefern.`

### F-2.18-5 — `package-info.java` listet `InqDecorator`-Adapter nicht auf

- **Bereich:** Drift / Doku-Konsistenz.
- **Priorität:** nachrangig.
- **Beobachtung:** `eu/inqudium/aspect/pipeline/package-info.java` zählt das Modul-API auf, listet
  aber weder `ElementLayerProvider` noch `AsyncElementLayerProvider` noch `AspectPipelineTerminal`
  noch `HybridAspectPipelineTerminal` auf. Diese vier Klassen sind seit 0.7/0.8 Teil des öffentlichen

- **Begründung der Priorität:** nachrangig — Modul-Javadoc ist kein Hot-Path und wird selten als
  primäre Referenz genutzt (Klassen-Javadoc und User-Guide sind die Hauptkanäle).
- **Routing-Vorschlag:** Trivial-Edit, kann gleichzeitig mit F-2.18-4 erfolgen. `TODO.md`-Eintrag
  oder kleine Patch-Aufgabe.

## Empfehlung für nachfolgende Sub-Steps

Die 2.18-Bewertung legt das in REFACTORING.md skizzierte **Resultat 1** nahe: "Aspekt-Pipeline ist
sauber kompatibel auf dem synchronen Pfad, kleine Test-Lücken; weiter zu 2.19".

Konkret:

1. **Kein eigener Sub-Step zwischen 2.18 und 2.19.** Die kritischsten Findings (F-2.18-1, F-2.18-2)
   betreffen den asynchronen Pfad und sind an einen ohnehin geplanten async-`InqBulkhead`-ADR
   gebunden. Eine Hybrid-Terminal-Lockerung als isolierte Aufgabe wäre außerhalb des
   Bulkhead-Refactor-Scopes.
2. **F-2.18-3 in 2.20 absorbieren.** Die Test-Lücke gehört strukturell zu den dort bereits
   gelisteten Szenarien (Wrapper-through-lifecycle, Wrapper/Proxy gegen echten Bulkhead). Konkrete
   Test-Klassen-Vorschläge in F-2.18-3 oben.
3. **F-2.18-4 und F-2.18-5 als Doku-Patches mit F-2.18-3 koppeln.** Die Javadoc-Beispiele werden
   sinnvoll, sobald die Tests in 2.20 echte Code-Form zeigen; `package-info.java` profitiert von
   derselben Gelegenheit.
4. **F-2.18-1 / F-2.18-2 in den ADR-Audit-Backlog.** Sie sind Entwurfsfragen, keine
   Implementations-Schulden.

Nach 2.18 wie geplant zu 2.19 (Spring-Integration) weitergehen.
