# 2.19 Spring and Spring Boot Integration Audit

**Scope:** Phase-2 sub-step 2.19 of `REFACTORING.md`. The second of the three integration audits
that follow ADR-033 closure. Verifies that the Spring integration in `inqudium-spring` and the
Spring Boot starter in `inqudium-spring-boot` are consistent with the new bulkhead architecture
(`InqBulkhead<A, R>` as `BulkheadHandle<ImperativeTag>`, `InqElement`, `InqDecorator<A, R>`).

**Method:** Inventory, then assessment against the four audit areas (2.19.2 — bean wiring and
`InqBulkhead` acceptance, 2.19.3 — lifecycle-phase compatibility, 2.19.4 — test coverage and
isolation). No code changes.

**Build status:** `mvn verify` across the full reactor — green.
Spring modules:

| `inqudium-spring` (sum across nested suites of `InqShieldAspectPlainSpringTest`) | 14 | 0 | 0 | 0 |
| `inqudium-spring-boot` (`InqSpringBootIntegrationTest` + `InqCustomRegistryTest`) | 28 | 0 | 0 | 0 |

## 2.19.2 — Bean wiring and `InqBulkhead` acceptance

This area combines the spec's "BulkheadHandle-Akzeptanz und Bean-Wiring" questions because the
production code couples them: bean type, naming, property binding, and conditional gating are all
properties of the single `InqAutoConfiguration` class plus the runtime cast inside
`InqShieldAspect`.

### Bean publication for bulkheads

There is no Spring-side bean definition for bulkheads in either module. The auto-configuration
does **not** produce `BulkheadHandle<ImperativeTag>` beans, nor does it produce `InqBulkhead`
beans, nor does it bridge the `inqudium-config` runtime DSL to Spring at all.

What it does instead: it expects the user to define **arbitrary** `InqElement` beans (any
implementation, sync or async, real bulkhead or stub) and collects them into an
`InqElementRegistry` keyed by `InqElement#name()`. The element instance and its API surface are
the user's responsibility. The auto-configuration is bulkhead-agnostic.

### Bean type

No type-tagged read surface is exposed. Beans flow through the registry as bare `InqElement`. The
sync chain in `InqShieldAspect:254` casts elements to `InqDecorator<Void, Object>`; the async
chain at `:265` casts to `InqAsyncDecorator<Void, Object>`. Neither cast is checked at registration
time.

| Bean type expected to satisfy a Spring-mediated method | Sync method | Async method (`CompletionStage` return) |
|-------------------------------------------------------|-------------|------------------------------------------|
| `InqBulkhead<Void, Object>` (ADR-033 Stage 3-built) | accepted (`InqBulkhead implements InqDecorator<A, R>` per `InqBulkhead.java:56`) | **`ClassCastException` at runtime** — `InqBulkhead` does not implement `InqAsyncDecorator` (deliberate, per `InqBulkhead.java:46-47` and `:195` TODO). The cast on line 265 fails. |
| Bean implementing only `InqDecorator` | accepted | `ClassCastException` |
| Bean implementing only `InqAsyncDecorator` | `ClassCastException` | accepted |
| Bean implementing both (the test stubs) | accepted | accepted |

This is structurally the same gap as F-2.18-1 (the AspectJ `HybridAspectPipelineTerminal`'s eager
both-or-nothing validation), surfaced through Spring's runtime cast instead of an eager
constructor cast: a real `InqBulkhead` mounted under `@InqBulkhead("name")` on a method returning
`CompletionStage` will throw `ClassCastException` from the aspect's async branch. Sync callers are
unaffected.

### Naming convention

The lookup key is `InqElement#name()`, populated by the user as the bulkhead's identity. Spring
bean ids are not used for lookup (the auto-configuration discards the bean name and uses the
element's `name()`). This is consistent with the new architecture: `InqBulkhead` exposes its own
name via `BulkheadHandle.name()` (returned by the `name()` accessor inherited from `InqElement`),
and that is the same string the registry stores. **Compatible.**

### Property binding (`@ConfigurationProperties` / `application.properties`)

There is **no `application.properties` schema and no `@ConfigurationProperties` class** in
`inqudium-spring-boot`. Bulkhead configuration is expected to be done programmatically via
`Inqudium.configure()...build()` inside a user-defined `@Bean` method that returns the resulting
`InqBulkhead` (or any other `InqElement`). The auto-configuration neither reads
`inqudium.bulkhead.<name>.maxConcurrentCalls`-style properties nor maps them onto `BulkheadSnapshot`
/ `BulkheadStrategyConfig`.

This means there is **no property-schema drift to detect** because there is no property schema in
the first place. The REFACTORING.md spec for 2.19 lists "Configuration properties
(application.properties / application.yml) map to the new configuration shape" as something to
verify; the answer is "the surface does not exist". This is a deliberate omission in the current
state of the codebase, not drift — but it is worth recording as a Finding because the
documentation in `docs/user-guide/pipeline-developers-guide.md:260` ("Spring Boot
(auto-configuration) — everything is automatic") sets an expectation that no property surface
fulfils.

### `@Conditional` behaviour

| Annotation | On | Effect | Verdict |
|------------|----|--------|---------|
| `@AutoConfiguration` | class | marks the class for ordering relative to other auto-configurations; activation still requires registration in `META-INF/spring/...AutoConfiguration.imports` (Spring Boot 3+) | see F-2.19-1 below |
| `@ConditionalOnMissingBean` | `inqElementRegistry(...)` method | skips registry bean if a user-supplied `InqElementRegistry` already exists | **kompatibel** |
| `@ConditionalOnMissingBean` | `inqShieldAspect(...)` method | skips aspect bean if a user-supplied `InqShieldAspect` already exists | **kompatibel** |
| (no `@ConditionalOnClass`) | — | the auto-configuration fires unconditionally as long as it is registered, even if `spring-aop` / `aspectjweaver` / `inqudium-imperative` is missing from the runtime classpath | **lückenhaft** — see F-2.19-3 below |

### Conclusion of 2.19.2

The Spring modules are paradigm-agnostic by construction (they only know `InqElement`), so the
"BulkheadHandle acceptance" question reduces to: does the Spring-mediated path accept an
`InqBulkhead` produced via `Inqudium.configure()`? **Yes for sync methods, no for async methods**,
mirroring the F-2.18-1 gap — `InqShieldAspect` casts to `InqAsyncDecorator` for any
`CompletionStage`-returning method, and `InqBulkhead` does not implement that interface.

## 2.19.3 — Lifecycle-phase compatibility

### Spring lifecycle and Inqudium lifecycle are decoupled

The Spring modules contain **zero lifecycle hooks**: no `@PostConstruct`, no `@PreDestroy`, no
`InitializingBean`, no `DisposableBean`, no `SmartLifecycle`, no `ApplicationListener<…>`, no
`ContextClosedEvent` handling. (Verified by grep across `inqudium-spring/src/main` and
`inqudium-spring-boot/src/main`.) This means:

- **Bean creation does not transition any bulkhead from cold to hot.** `InqAutoConfiguration` only
  reads `name()` from each `InqElement` and stores the reference in `InqElementRegistry`. It does
  not call `bulkhead.execute(...)`. Whatever phase the bulkhead is in at the moment Spring asks for
  the bean is the phase it stays in until the user code calls `execute(...)` for the first time
  (which is what triggers cold-to-hot, per the standard `ImperativeLifecyclePhasedComponent`
  contract). The first such call typically happens through the aspect path on the first business
  request — exactly as with non-Spring usage. **Compatible.**
- **Spring shutdown does not call `markRemoved()`.** When `ApplicationContext.close()` runs, Spring
  disposes beans according to their declared destruction callbacks. `InqBulkhead` does not
  implement `AutoCloseable`/`Closeable` (its lifecycle method is `markRemoved()`, which has no
  destruction-protocol equivalent that Spring would discover automatically). The auto-configuration
  does not register a destruction callback (e.g. `@Bean(destroyMethod = "markRemoved")`) and does
  not listen for `ContextClosedEvent`. Result: on context shutdown, bulkheads remain in their
  current phase (typically `HotPhase`); the strategy holds whatever resources it allocated; the
  JVM exits and reclaims everything. No leak in the typical "shutdown the JVM" case. In the
  "context restart" case (rare in production but common in tests with multiple `@SpringBootTest`
  classes), the previous bulkhead instances become unreachable but were never told they were being
  removed. Memory-wise this is benign because `InqBulkhead` does not register external listeners
  that would keep it alive. **lückenhaft, not blocking** — see F-2.19-4.

### Hot-swap visibility for Spring-consuming components

The aspect's per-`Method` cache stores `InqElement` references (more precisely: the cached
`Function<JoinPointExecutor<Object>, JoinPointExecutor<Object>>` holds `element` in a lambda
closure, captured by the lambda inside `buildSyncChainFactory` at line 253). These references are
to the live bean — not to a snapshot of the bulkhead's internal phase or strategy. Each invocation
that takes the cached path goes:

```
@Around → around(pjp)
       → cache.get(method).syncFactory.apply(terminal)
            → composed lambda chain
                 → for each element, call element.decorateJoinPoint(...) per call
                      → element.execute(c, ca, a, next)        ← @Override on InqBulkhead (final)
                            → ImperativeLifecyclePhasedComponent.execute
                                 → phase.get()      (AtomicReference, volatile read)
                                 → phase.execute(...)
                                      → BulkheadHotPhase.execute → strategy (volatile)
```

`decorateJoinPoint` is a default method on `InqDecorator` that wraps `JoinPointExecutor.proceed()`
into a `LayerAction` and calls `execute(...)` on the same instance. The element identity is stable
across the bean lifetime; `phase` and `strategy` are re-read per call. **Compatible.**

| Lifecycle event | Spring-mediated path behaviour |
|-----------------|--------------------------------|
| Cold-to-hot transition | First aspect-mediated call hits `InqBulkhead.execute` → `phase.get()` returns `ColdPhase` → cold phase runs the CAS, fires `ComponentBecameHotEvent`, delegates to the hot phase. Subsequent aspect-mediated calls observe `HotPhase`. **Compatible.** No cached pre-hot reference exists in the aspect. |
| Strategy hot-swap (ADR-032) | Snapshot listener installed by `BulkheadHotPhase.afterCommit` writes the new `BulkheadStrategy` to the volatile `strategy` field. The next aspect-mediated call enters `BulkheadHotPhase.execute`, which re-reads `strategy`. **Compatible.** The aspect cache holds the bean reference, not a `BulkheadStrategy` or `BulkheadHotPhase` reference. |
| Removal (`markRemoved()`) | `phase` becomes `RemovedPhase`. Next aspect-mediated call → `ImperativeLifecyclePhasedComponent.execute` checks `if (current instanceof RemovedPhase) throw new ComponentRemovedException(...)`. The exception propagates back through the layer chain, out of the aspect's `@Around`, through Spring AOP's invocation handler, to the caller. **Compatible.** Spring AOP surfaces the original throwable type unchanged. |
| Bean replacement via `InqElementRegistry.register(name, newElement)` after the aspect cache has been built for a method that uses that name | Aspect cache still holds the **old** element reference (the cache key is `Method`, the lookup happens at first invocation). Subsequent calls bypass the new element. **Edge case** — see F-2.19-5. Not strictly a Spring problem; same drift exists for any user that re-registers an element after a method has been intercepted. |

### Aspect-cache surfaces inspected, no cached lifecycle state found

- `InqShieldAspect.cache`: `ConcurrentHashMap<Method, CachedPipeline>`; `CachedPipeline` holds an
  `async` flag plus one of two `Function` references that compose a `JoinPointExecutor` chain. The
  `Function` closures over `element` references obtained from the registry through `PipelineFactory`.
  The element is the live bean. No phase, strategy, or snapshot reference is cached.
- `InqElementRegistry.elements`: `ConcurrentHashMap<String, InqElement>`. Stable references to
  beans; no per-element snapshot cache.

### Conclusion of 2.19.3

The Spring proxy / aspect path routes every invocation through a method reference on the live
`InqElement`, which then reads volatile state per call. Nothing is cached at a level below the
element's own indirection. Cold-to-hot, strategy hot-swap, and removal are structurally
compatible. The two notable gaps are (1) Spring shutdown does not signal `markRemoved()`
(F-2.19-4), and (2) registry replacement is invisible to the aspect cache (F-2.19-5).

### Spring context isolation

- The two `inqudium-spring-boot` test classes use **different `classes` attributes** on
  `@SpringBootTest`, so the Spring TestContext cache treats them as separate contexts. No
  classpath-scan cross-pollution between them.
- `InqSpringBootIntegrationTest` activates auto-configuration via **`@Import(InqAutoConfiguration.class)`**
  on its `TestApplication`. That works regardless of whether the auto-configuration descriptor
  exists.
- `InqCustomRegistryTest` activates auto-configuration via `@EnableAutoConfiguration` on its
  `CustomRegistryConfig`. **This relies on the auto-configuration descriptor**, which is missing
  (see F-2.19-1). Consequence: `@EnableAutoConfiguration` does **not** discover
  `InqAutoConfiguration` in the test, so the test never actually exercises the
  `@ConditionalOnMissingBean` skip path — the auto-configured registry bean is never even
  evaluated. Both test assertions still pass, but for the wrong reason: they pass because the user
  registry is the only `InqElementRegistry` in the context, regardless of whether the
  auto-configuration would have skipped or overwritten it. See F-2.19-2.
- The `InqShieldAspectPlainSpringTest` is in `inqudium-spring`, where Spring Boot is not on the
  classpath at all — there is nothing to pollute via `@SpringBootApplication` classpath scanning.
- The `CLAUDE.md` warning ("Spring Boot often captures too much information through classpath
  scanning") applies less here than it could because the Spring-Boot test classes pin `classes =
  ...` explicitly. The `@SpringBootTest` does not auto-detect by walking up the package hierarchy,
  because no `*ApplicationTest`-named bootstrap class is present at a higher level.

### Test gaps relative to 2.19's audit areas

- No test mounts a real `Inqudium.configure()`-built `InqBulkhead` as a Spring `@Bean` and exercises
  the aspect-mediated path. (Closest cousin: `Adr033Stage3ClosureTest` in `inqudium-aspect`, but
  that is structural-compile-only and runs without Spring.)
- No test exercises the cold-to-hot transition through the Spring-mediated aspect path on a real
  bulkhead.
- No test triggers a strategy hot-swap on a Spring-published bulkhead and asserts that the next
  aspect-mediated call uses the new strategy.
- No test calls a Spring-published bulkhead after `markRemoved()` and asserts
  `ComponentRemovedException` propagation through Spring AOP.
- No test exercises Spring `ApplicationContext.close()` on a context that owns a hot bulkhead and
  asserts a defined post-shutdown state (relates to F-2.19-4).
- No test pins the async-bulkhead `ClassCastException` symptom (F-2.19-6) — neither as a "known
  symptom" assertion nor as a guard rail.

This is the same shape as the 2.18.4 gap on the AspectJ side: the structural compatibility shown
by 2.19.2 and 2.19.3 makes these tests writeable today; the absence is a routing question, not a
blocker.

## Findings

### F-2.19-1 — Auto-configuration descriptor is missing from `inqudium-spring-boot`

- **Bereich:** 2.19.2 (bean wiring / auto-config activation), 2.19.4 (test isolation).
- **Priorität:** kritisch.
- **Beobachtung:** `inqudium-spring-boot/src/main/resources/` enthält **nur** `.gitkeep`. Es gibt
  weder `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`
  (Spring Boot 3+/4+ Mechanismus) noch das ältere `META-INF/spring.factories`. Das gepackte
  `inqudium-spring-boot-0.7.0-SNAPSHOT.jar` wurde geprüft und enthält dieselbe Lücke. Folge:
  ein Anwender, der nur `inqudium-spring-boot` zu seiner Spring-Boot-Anwendung hinzufügt,
  erhält **keine automatische Aktivierung** von `InqAutoConfiguration`. Spring Boot's
  `@EnableAutoConfiguration` (implizit aktiviert durch `@SpringBootApplication`) liest die
  `AutoConfiguration.imports`-Dateien jedes Classpath-Eintrags; ohne die Datei wird die Klasse
  nicht entdeckt. Das `@AutoConfiguration`-Annotation an der Klasse beeinflusst nur die
  Reihenfolge zwischen bereits entdeckten Auto-Configs, nicht die Entdeckung selbst.
- **Begründung der Priorität:** kritisch, weil das die zentrale öffentliche Aussage des Moduls
  bricht. `pipeline-developers-guide.md:260-269` sagt wörtlich: "Add `inqudium-spring-boot` —
  everything is automatic." Tatsächlich ist nichts automatisch; der Anwender muss
  `@Import(InqAutoConfiguration.class)` selbst schreiben — exakt das, was der Integrationstest
  `InqSpringBootIntegrationTest` als Workaround bei `TestApplication` tut, ohne den Workaround
  als solchen zu kennzeichnen. Auch der Custom-Registry-Test
  (`InqCustomRegistryTest`) baut auf der nicht-existenten Autodiscovery auf — er besteht
  trivialerweise aus einem anderen Grund (siehe F-2.19-2).
- **Routing-Vorschlag:** Eigener kleiner Sub-Step direkt nach 2.19, **vor** 2.20. Die Datei zu
  ergänzen ist trivial:
  ```
  inqudium-spring-boot/src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
  ```
  mit Inhalt:
  ```
  eu.inqudium.spring.boot.InqAutoConfiguration
  ```
  Plus: `InqCustomRegistryTest` gegen den (jetzt funktionierenden) Auto-Discovery-Pfad
  korrigieren — er sollte zusätzlich pinnen, dass die auto-konfigurierte Registry tatsächlich
  durch die User-Registry ersetzt wird (z. B. durch ein zweites `InqElement`-Bean im Test, das
  *nicht* in der User-Registry landet, aber landen würde, wenn die Auto-Config feuern würde).
  Das ist genau der "wird die `@ConditionalOnMissingBean` tatsächlich getriggert"-Test, der
  heute fehlt. Diesen Sub-Step nicht in 2.20 absorbieren, weil 2.20 ein integrationstests-modul
  ist; das Fix gehört zur Produktion von `inqudium-spring-boot` selbst.

### F-2.19-2 — `InqCustomRegistryTest` testet `@ConditionalOnMissingBean` nicht wirklich

- **Bereich:** 2.19.4 (test correctness).
- **Priorität:** wichtig (folgt aus F-2.19-1).
- **Beobachtung:** Der Test verwendet `@EnableAutoConfiguration` auf `CustomRegistryConfig`. Da
  die `AutoConfiguration.imports`-Datei fehlt (F-2.19-1), entdeckt `@EnableAutoConfiguration`
  `InqAutoConfiguration` nicht, und der Test exerziert in Wirklichkeit *gar keine
  Auto-Konfiguration* — er prüft nur, dass die manuell eingebrachte `InqElementRegistry` mit
  einem Element aus dem Context herausgeholt werden kann. Beide Assertions
  (`registry.size()==1`, `!registry.contains("orderCb")`) bestehen aus dem trivialen Grund, dass
  niemand etwas zur Registry hinzugefügt hat außer dem Test selbst. Der Name des Tests
  ("Custom registry overrides auto-configuration") und die Kommentare im Test suggerieren das
  Gegenteil.
- **Begründung der Priorität:** wichtig — der Test ist heute Green-by-coincidence. Sobald
  F-2.19-1 behoben ist, würde der Test echtes Verhalten prüfen, müsste aber stärker werden, um
  die Override-Semantik tatsächlich zu pinnen. Bei Reverts oder Refactorings, die
  `@ConditionalOnMissingBean` versehentlich entfernen, würde der heutige Test das nicht fangen.
- **Routing-Vorschlag:** zusammen mit F-2.19-1 lösen. Konkrete neue Test-Form:
  - Test-Setup: `@SpringBootTest(classes = CustomRegistryConfig.class)` mit
    `CustomRegistryConfig` enthält **drei** Beans:
    1. eine User-Registry mit einem Element `customCb`,
    2. ein zusätzliches `@Bean InqElement` namens `autoDiscovered`,
    3. (impliziert über klassenpfad-discovery) das `InqShieldAspect`-Bean.
  - Nach Initialisierung soll die User-Registry intakt sein (`size==1`, nur `customCb`), aber
    das `autoDiscovered`-Element ist über Spring direkt abrufbar — es ist nur nicht in der
    Registry. Damit ist der Override-Pfad deterministisch.
  - Zweiter Test: `inqShieldAspect`-Bean existiert ebenfalls (verifiziert, dass die zweite
    Auto-Config-Methode bei vorhandener Registry weiterlaufen kann, wenn kein User-Aspekt
    bereitsteht).

### F-2.19-3 — Auto-Configuration ohne `@ConditionalOnClass`-Gating

- **Bereich:** 2.19.2 (conditional behaviour).
- **Priorität:** nachrangig.
- **Beobachtung:** `InqAutoConfiguration` hat keine Klassen-Conditionals. Wenn ein Anwender das
  Modul auf den Classpath legt, aber etwa `aspectjweaver` (transitiv über `inqudium-spring`) oder
  `inqudium-imperative` (Quelle für `InqAsyncDecorator`) durch ein eigenes Exclusion-Setup
  entfernt, feuert die Auto-Config trotzdem und scheitert dann erst beim Bean-Bau (z. B. weil
  `InqShieldAspect` zur Klassen-Resolutionszeit `InqAsyncDecorator` benötigt, sobald die
  `buildAsyncChainFactory`-Methode geladen wird). Best-practice-Spring-Boot gating wäre:
  ```java
  @ConditionalOnClass({InqShieldAspect.class, ProceedingJoinPoint.class})
  ```
  oder feiner pro Bean.
- **Begründung der Priorität:** nachrangig — das Standardszenario (Anwender bringt das
  Spring-Boot-Starter-Paket plus die Standardabhängigkeiten) funktioniert. Nur das
  exklusionsfreudige Sub-Szenario triggert das Problem, und das produziert eine eindeutige
  `NoClassDefFoundError`. Kein Datenverlust, kein silent-failure.
- **Routing-Vorschlag:** `TODO.md`-Eintrag, lösbar gemeinsam mit F-2.19-1, weil beide zur
  Auto-Config-Hygiene gehören.

### F-2.19-4 — Spring-Context-Shutdown ruft `markRemoved()` nicht auf

- **Bereich:** 2.19.3 (lifecycle).
- **Priorität:** wichtig (für Test-Isolation und Multi-Context-Anwendungen), nachrangig (für
  einfache JVM-shutdown-Szenarien).
- **Beobachtung:** Weder `InqAutoConfiguration` noch `InqShieldAspect` reagiert auf Spring's
  Lifecycle-Events. Wenn `ApplicationContext.close()` ausgeführt wird, bleibt jeder hot
  `InqBulkhead` in `HotPhase` und seine Strategy-Resources werden nicht freigegeben. Kein
  `DisposableBean`, kein `@PreDestroy`, kein `ContextClosedEvent`-Listener, kein
  `destroyMethod = "markRemoved"` an einer `@Bean`-Annotation.
- **Begründung der Priorität:** wichtig im Test-Kontext, weil `@SpringBootTest` Context-Caching
  betreibt und ein Context-Reset (etwa über `@DirtiesContext`) heute die Bulkheads zurücklässt.
  Im Production-Code mit JVM-Shutdown ist es i. d. R. unkritisch, weil der Prozess endet, bevor
  Resource-Leaks materialisieren. In `Multi-Tenant`-Szenarien (mehrere Spring-Contexts in einer
  JVM) wäre es ein echter Leak.
- **Routing-Vorschlag:** weder Sub-Step noch 2.20-Absorption ist sofort fällig — das ist eine
  Design-Frage, die zum ADR-Audit-Backlog gehört. Frage: soll
  `InqAutoConfiguration` jeden `InqElement`-Bean durch ein `BeanFactoryPostProcessor`-Pendant
  registrieren, das beim Context-Shutdown `markRemoved()` aufruft? Oder soll der Anwender das
  selbst tun (über `@Bean(destroyMethod = "markRemoved")`)? Auch `InqElement` selbst kennt heute
  keine Standard-Lifecycle-Methode `markRemoved()` — die ist auf `InqBulkhead`-Ebene definiert.
  Eine generische Lösung erfordert entweder einen Lifecycle-SPI auf `InqElement` oder ein
  paradigm-spezifisches Bridging. **Eintrag in den ADR-Audit-Backlog** als
  "Spring-Lifecycle-Bridging".

### F-2.19-5 — Aspect-Cache ist gegen Registry-Replacement nicht invalidiert

- **Bereich:** 2.19.3 (hot-swap visibility).
- **Priorität:** nachrangig.
- **Beobachtung:** `InqShieldAspect` cached die zusammengesetzte Pipeline pro `Method` und hält
  in den Lambda-Closures direkte `InqElement`-Referenzen. Wenn ein Anwender die Registry mit
  `registry.register(name, newElement)` *nach* dem ersten Aufruf einer durch den Aspekt
  abgedeckten Methode aktualisiert, sieht der Aspekt weiterhin die alte Element-Instanz. Das ist
  kein bulkhead-spezifisches Problem (snapshot-basierte Hot-Swaps ändern nicht die
  Element-Identität — der Aspekt-Cache funktioniert für sie korrekt) — es betrifft nur
  *Element-Replacement* in der Registry.
- **Begründung der Priorität:** nachrangig — Element-Replacement ist kein dokumentierter
  Anwendungsfall, weder im User-Guide noch in den Snapshot-/Hot-Swap-ADRs. Hot-Swap geht über
  Snapshots, nicht über Registry-Mutation. Der Fall müsste konstruiert werden, um zu greifen.
- **Routing-Vorschlag:** als `TODO.md`-Eintrag dokumentieren, falls explizite Registry-Mutation
  je dokumentiert wird. Bis dahin ist die heutige Semantik (Cache überlebt
  Registry-Replacement) konsistent mit der allgemeinen Aussage "Element-Replacement wird nicht
  unterstützt — Hot-Swap geht über Snapshots". Kein Sub-Step, keine 2.20-Absorption.

### F-2.19-6 — Async-Pfad mit `InqBulkhead` wirft `ClassCastException` zur Laufzeit

- **Bereich:** 2.19.2 (acceptance, async).
- **Priorität:** wichtig (für den async-Pfad; nicht-blockierend für aktuelle Nutzer).
- **Beobachtung:** `InqShieldAspect.buildAsyncChainFactory` (Zeile 258-267) cast jedes
  Pipeline-Element zu `InqAsyncDecorator<Void, Object>`. Eine `@InqBulkhead("name")`-Annotation
  auf einer Methode mit `CompletionStage`-Returntyp führt dazu, dass der Aspekt das
  registrierte `InqBulkhead`-Element für den Async-Branch verwendet — und `InqBulkhead`
  implementiert `InqAsyncDecorator` nicht (Javadoc Zeile 46-47, `TODO` Zeile 195). Resultat:
  `ClassCastException` beim ersten Aufruf der Methode.
- **Begründung der Priorität:** wichtig, nicht kritisch — analog zu F-2.18-1. Die synchrone
  Bulkhead-Spring-Integration funktioniert (sync-Methoden mit `@InqBulkhead` ohne
  `CompletionStage`-Returntyp sind unproblematisch), und der Refactor-Scope deckt async-Bulkhead
  ohnehin erst über einen separaten ADR ab. Aber: jeder Async-Spring-Setup, der heute mit dem
  deprecated `Bulkhead`/`ImperativeBulkhead` arbeitet, kann nicht migriert werden, ohne
  `@InqBulkhead` aus async-Methoden zu entfernen.
- **Routing-Vorschlag:** an den ADR-Audit-Backlog weitergeben, gemeinsam mit F-2.18-1 und
  F-2.18-2 als ein Block ("Async-Bulkhead und Aspect-/Spring-Integration"). Eigentliche
  Auflösung erfolgt mit dem async-`InqBulkhead`-ADR. Optional: ein defensiver Check im
  `buildCachedPipeline` (vor dem Cache-Eintrag), der für jedes Async-Element prüft, ob es
  `InqAsyncDecorator` implementiert, und sonst bei der Pipeline-Konstruktion fail-fast statt
  beim ersten Call ist sinnvoll, aber das ist Material für den ADR-Audit. Kein eigener Sub-Step
  innerhalb dieses Bulkhead-Refactors.

### F-2.19-7 — Spring-Tests gegen echte `InqBulkhead` fehlen

- **Bereich:** 2.19.4 (test coverage).
- **Priorität:** wichtig (analog zu F-2.18-3 / Audit-Finding 2.17.4).
- **Beobachtung:** Kein Spring-Test exerziert einen `Inqudium.configure()`-gebauten Bulkhead. Alle
  drei Test-Klassen (`InqShieldAspectPlainSpringTest`, `InqSpringBootIntegrationTest`,
  `InqCustomRegistryTest`) verwenden Stub-Decorator (`TracingElement`, `DualTracingElement`,
  `StubElement`). Die in 2.19.3 strukturell validierten Lifecycle-Eigenschaften
  (cold-to-hot, hot-swap, removal) sind end-to-end durch den Spring-Pfad nicht gepinnt.
- **Begründung der Priorität:** wichtig, weil:
  1. Der strukturelle Beweis in 2.19.3 ist ein Code-Review-Argument, kein Regressionsschutz.
     Eine Refaktorisierung an `BulkheadHotPhase`, `ImperativeLifecyclePhasedComponent` oder
     `InqShieldAspect.buildSyncChainFactory` könnte unbemerkt das Lifecycle-Verhalten durch den
     Spring-AOP-Pfad brechen.
  2. ADR-033 hat die strukturelle Voraussetzung (`InqBulkhead` ⇒ `InqDecorator`) gerade erst
     geschaffen — wie auch bei F-2.18-3 ist die Stub-Tendenz aus der Pre-ADR-033-Ära Pragmatismus,
     nicht Designprinzip.
  3. Die Test-Form ist analog zu F-2.18-3 (Aspect-Pfad gegen echten Bulkhead); wenn 2.20 ohnehin
     dieses Test-Material aufnimmt, ist die Inklusion der Spring-Pfade kostengünstig.
- **Routing-Vorschlag:** in 2.20 (`inqudium-bulkhead-integration-tests`) absorbieren, parallel zu
  den dort bereits gelisteten "Wrapper compatibility across cold-to-hot transition,
  strategy hot-swap, and structural removal" und den "Wrapper and proxy tests run against real
  bulkheads" Findings sowie F-2.18-3. Konkrete neue Test-Klassen in dem Modul:
  - `BulkheadSpringPlainLifecycleTest` (in einem `spring/`-Unterpaket des 2.20-Moduls): drei
    Szenarien analog zu F-2.18-3 (cold-to-hot durch Spring-AOP, strategy hot-swap, removal),
    aber durch den `InqShieldAspect`-Pfad in einem plain-Spring-Context (`@ContextConfiguration`).
  - `BulkheadSpringBootLifecycleTest`: dieselben Szenarien, aber mit echtem `@SpringBootTest` und
    funktionierender Auto-Konfiguration (setzt F-2.19-1 voraus).
  - Optional: `BulkheadSpringShutdownTest` — pinnt das aktuelle "Shutdown ruft markRemoved()
    nicht auf"-Verhalten (F-2.19-4) als Regressionsbarriere, bis F-2.19-4 eine
    Design-Entscheidung getroffen hat.
  Voraussetzung: das 2.20-Modul muss Maven-Abhängigkeiten zu `inqudium-spring`,
  `inqudium-spring-boot`, `inqudium-imperative`, `inqudium-config` aufnehmen.

### F-2.19-8 — Javadoc-Beispiele in `InqAutoConfiguration` zeigen veraltete Konstruktoren

- **Bereich:** Drift / Doku-Konsistenz.
- **Priorität:** nachrangig.
- **Beobachtung:** `InqAutoConfiguration.java:35-43` zeigt:
  ```java
  @Bean
  public CircuitBreaker paymentCb() {
      return CircuitBreaker.of(config);  // name() returns "paymentCb"
  }

  @Bean
  public Retry paymentRetry() {
      return Retry.of(config);  // name() returns "paymentRetry"
  }
  ```
  Weder `CircuitBreaker.of(config)` noch `Retry.of(config)` existiert in der neuen Architektur:
  - `Inqudium.configure().imperative(im -> im.bulkhead(...)).build()` ist der einzige Pfad,
    einen `InqBulkhead` zu materialisieren.
  - Ein `CircuitBreaker`-/`Retry`-API in der gezeigten Form existiert (noch) nicht — diese
    Komponenten sind außerhalb des aktuellen Bulkhead-Refactors.
  Außerdem suggerieren die Beispiele, dass `name()` aus dem `@Bean`-Methodennamen abgeleitet
  wird; tatsächlich kommt der Name aus der Komponentenkonfiguration im DSL.
- **Begründung der Priorität:** nachrangig — die Javadoc-Beispiele dokumentieren ein API, das
  nie für `InqBulkhead` existiert hat. Sie verwirren neue Leser, blockieren aber niemanden. Der
  konstruktive Umbau der Beispiele wird sinnvoll, sobald F-2.19-7 / 2.20 eine echte
  Spring-Boot-Test-Klasse mit echtem `Inqudium.configure()`-Bulkhead bereitstellt — dann gibt
  es eine Referenz zum Verlinken.
- **Routing-Vorschlag:** Doku-Korrektur als trivialer Edit zulässig. Nicht jetzt vornehmen,
  sondern *gemeinsam* mit F-2.19-7 in 2.20 — analog zur Empfehlung F-2.18-4. Alternativ
  `TODO.md`-Eintrag `TODO(2.20): Javadoc-Beispiele in InqAutoConfiguration auf
  Inqudium.configure() umstellen, sobald die Spring-Integration-Tests Referenzcode liefern.`

### F-2.19-9 — Property-Schema fehlt vollständig

- **Bereich:** Drift / Konzeptionelle Lücke.
- **Priorität:** nachrangig (für den aktuellen Refactor) / wichtig (für die User-Guide-Aussage).
- **Beobachtung:** `inqudium-spring-boot` enthält weder `@ConfigurationProperties`-Klassen noch
  `additional-spring-configuration-metadata.json` noch eine `application.properties`-Beispieldatei.
  REFACTORING.md 2.19 erwartet "Configuration properties (application.properties /
  application.yml) map to the new configuration shape" — die Antwort lautet: dieses Mapping
  existiert nicht. Der User-Guide
  (`pipeline-developers-guide.md:262`) schreibt "Add `inqudium-spring-boot` — everything is
  automatic", was leicht so gelesen werden kann, dass auch eine Property-Schema-Aktivierung
  erwartet wird. Das ist nicht der Fall — der Anwender muss bulkhead-Konfiguration heute
  programmatisch in `@Bean`-Methoden via `Inqudium.configure()` aufbauen.
- **Begründung der Priorität:** Innerhalb des Bulkhead-Refactors ist das nachrangig — die
  Snapshot-/Patch-Architektur ist die Quelle der Wahrheit, und ein Property-Mapping wäre eine
  zusätzliche Schicht obenauf, kein Drift gegen die neue Architektur. Wichtig wird es, sobald
  Spring-Boot-Anwender ein property-getriebenes Setup erwarten — dann ist das ein vollwertiges
  Feature und braucht ein eigenes ADR.
- **Routing-Vorschlag:** ADR-Audit-Backlog. Konkret: **kein Sub-Step in diesem Refactor**, weil
  das Property-Mapping orthogonal zum Bulkhead-Refactor steht und ohnehin alle Komponenten
  betreffen würde, nicht nur den Bulkhead. Optional: User-Guide-Korrektur ("everything is
  automatic — sobald Sie Ihre Elemente als `@Bean` definieren") als Doku-Patch zusammen mit
  F-2.19-8.

## Empfehlung für nachfolgende Sub-Steps

Die 2.19-Bewertung legt eine Mischung aus **Resultat 2** und **Resultat 3** aus REFACTORING.md
nahe:

- **Resultat 2 ist teilweise zutreffend**, weil F-2.19-1 (fehlende AutoConfiguration.imports)
  ein konkretes, lokal lösbares Drift-Problem ist, das einen kleinen, klar abgegrenzten
  Sub-Step rechtfertigt. Das ist kein "substantielles" Drift im Sinne eines mehrtägigen
  Refactors, aber es ist ein produktiver Bug, der die zentrale öffentliche
  Auto-Config-Aussage bricht.
- **Resultat 3 ist für die Bulkhead-Architektur-Frage zutreffend**: die Spring-Integration ist
  von Phase 2 (Snapshot/Patch/Lifecycle) strukturell unberührt — alle Tests laufen heute
  durch generische `InqElement`/`InqDecorator`/`InqAsyncDecorator`-Verträge, und die
  paradigm-agnostische Schicht ist sauber.

Konkret:

1. **F-2.19-1 + F-2.19-2 + F-2.19-3 als kleiner Sub-Step `2.19a Spring-Boot Auto-Configuration
   Descriptor` direkt nach 2.19, vor 2.20.** Inhalt: AutoConfiguration.imports anlegen,
   `InqCustomRegistryTest` so umbauen, dass er den Override-Pfad echt prüft, und
   `@ConditionalOnClass` an `InqAutoConfiguration` ergänzen. Klein, lokal, keine
   Architektur-Implikationen. Der Build wird grün bleiben (die existierenden Tests verwenden
   `@Import`, das funktioniert auch nach dem Fix; der Custom-Registry-Test wird strikter).
2. **F-2.19-7 in 2.20 absorbieren.** Die Test-Lücke gehört strukturell zu den dort bereits
   gelisteten Szenarien (Wrapper-through-lifecycle, Wrapper/Proxy gegen echten Bulkhead,
   F-2.18-3). Konkrete Test-Klassen-Vorschläge in F-2.19-7 oben. Voraussetzung: F-2.19-1 muss
   vorher gelöst sein, sonst pollutiert der "kein Auto-Config-Discovery"-Bug die Tests.
3. **F-2.19-8 + F-2.19-9 (User-Guide-Hälfte) als Doku-Patches mit F-2.19-7 koppeln.** Die
   Javadoc-Beispiele und der User-Guide-Satz "everything is automatic" werden sinnvoll, sobald
   die Tests in 2.20 echte Code-Form zeigen.
4. **F-2.19-4 (Spring-Lifecycle-Bridging) und F-2.19-6 (Async-Bulkhead) und F-2.19-9
   (Property-Schema) in den ADR-Audit-Backlog.** Sie sind Entwurfsfragen, keine
   Implementations-Schulden.
5. **F-2.19-5 als `TODO.md`-Eintrag** für den Tag, an dem Element-Replacement-via-Registry
   dokumentiert oder unterstützt werden soll. Heute kein Handlungsbedarf.

Nach Erledigung des kleinen Sub-Steps `2.19a` wie geplant zu 2.20
(`inqudium-bulkhead-integration-tests`) weitergehen.
